# v0.6.0 Migration Guide: Timestamp Field Renames

#migration #breaking-change

> [!warning] Breaking Change
> v0.6.0 renames timestamp fields across `AlarmRecord`, `WriteCommand`, `ActionCommand`, and `CommandRecord`.
> Existing MongoDB documents must be migrated before upgrading. See the steps below.

## Why This Change

In earlier versions, timestamp fields had inconsistent names:
- `AlarmRecord` used `occurred_at` / `resolved_at`
- `CommandRecord` / `WriteCommand` / `ActionCommand` used `created_at`

`DataUploadManager` and `StateSyncManager` both used `timestamp` as their canonical field name. v0.6.0 aligns the alarm and command schemas to the same convention, making cross-collection queries and data pipeline code consistent.

---

## Breaking Changes

### AlarmRecord field renames

| Old field | New field | Notes |
|-----------|-----------|-------|
| `occurred_at` | `timestamp` | Alarm trigger time |
| `resolved_at` | `resolved_timestamp` | Alarm resolution time |

### WriteCommand / ActionCommand field renames

| Old field | New field |
|-----------|-----------|
| `created_at` | `timestamp` |

### CommandRecord field renames

| Old field | New field |
|-----------|-----------|
| `created_at` | `timestamp` |

---

## MongoDB Data Migration

Run these commands against your MongoDB instance **before** deploying v0.6.0 application code.

### Alarms collection

```javascript
db.alarms.updateMany({}, {
  $rename: {
    "occurred_at": "timestamp",
    "resolved_at": "resolved_timestamp"
  }
})
```

### Commands collection

```javascript
db.commands.updateMany({}, {
  $rename: {
    "created_at": "timestamp"
  }
})
```

### Index Rebuild

After renaming, drop and recreate indexes so queries remain performant:

```javascript
// Alarms — drop old indexes
db.alarms.dropIndex("occurred_at_-1")

// Alarms — create new indexes
db.alarms.createIndex({ "timestamp": -1 })
db.alarms.createIndex({ "resolved_timestamp": -1 }, { sparse: true })

// Commands — drop old index
db.commands.dropIndex("created_at_-1")

// Commands — create new index
db.commands.createIndex({ "timestamp": -1 })
```

> [!note] Index Names
> Your existing index names may differ depending on how they were created. Use `db.alarms.getIndexes()` to list current index names before dropping.

## Application Code Changes

Replace all usages in your application code:

### AlarmRecord

```python
# Before
record.occurred_at
record.resolved_at
AlarmRecord(occurred_at=datetime.now(), ...)

# After
record.timestamp
record.resolved_timestamp
AlarmRecord(timestamp=datetime.now(), ...)
```

### WriteCommand / ActionCommand

```python
# Before
command.created_at

# After
command.timestamp
```

### CommandRecord

```python
# Before
record.created_at

# After
record.timestamp
```

---

## Rollback

If you need to revert to v0.5.x after migrating documents, run the reverse rename:

```javascript
// Alarms
db.alarms.updateMany({}, {
  $rename: {
    "timestamp": "occurred_at",
    "resolved_timestamp": "resolved_at"
  }
})

// Commands
db.commands.updateMany({}, {
  $rename: {
    "timestamp": "created_at"
  }
})
```

> [!warning] Application Code Must Also Revert
> Rolling back MongoDB documents without also downgrading the application to v0.5.x will cause errors. Ensure both steps are done together.
